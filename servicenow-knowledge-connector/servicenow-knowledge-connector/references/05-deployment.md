# Deploy the ServiceNow Knowledge Copilot connector

Source: https://learn.microsoft.com/en-us/microsoft-365/copilot/connectors/servicenow-knowledge-deployment

## Prerequisites

| Role | Task |
|---|---|
| ServiceNow admin | Configure the environment |
| ServiceNow admin | Set up prerequisites |
| Microsoft 365 admin | Deploy the connector in the Microsoft 365 admin center |
| Microsoft 365 admin | Customize connector settings (optional) |

Before deploying:
- You're a Microsoft 365 admin.
- Access to a configured ServiceNow instance.
- REST API access enabled for required tables.
- ACLs configured to allow read access.
- ServiceNow instance URL identified.

## Deploy the connector

1. In the Microsoft 365 admin center, **Copilot > Connectors**.
2. Connectors tab > **Gallery** > **ServiceNow Knowledge**.

### Set display name

Identifies references in Copilot responses. Accept default or customize. Signifies trusted content and is used as a content source filter.

### Choose flow based on user criteria

- **Simple (default)**: advanced script-based user criteria NOT evaluated. Evaluates KB-level and article-level user criteria.
- **Advanced**: evaluates script-based user criteria via the Scripted REST API. Requires REST API setup.

### Set instance URL

Format: `https://<instance-name>.service-now.com`

### Choose authentication type

- **Federated Auth (recommended)** — uses a Microsoft app with OIDC; no client secret to store/rotate.
- **Basic authentication** — username/password of a ServiceNow account with knowledge role.
- **OAuth 2.0** — endpoint registration in ServiceNow.
- **Microsoft Entra ID OpenID Connect** — single-tenant Entra app + ServiceNow OAuth OIDC entity.

## Federated Auth (Federated Identity Credentials)

Prerequisites:
- ServiceNow admin with privileges to create OIDC providers and users.
- Microsoft Entra tenant ID (Directory ID).

### Step 1: Get the service principal object ID

Service Principal Object ID of the first-party connector application in your tenant (NOT a new app registration).

**Option A: PowerShell**
```powershell
Install-Module -Name Az -AllowClobber -Scope CurrentUser
Connect-AzAccount
Get-AzADServicePrincipal -ApplicationId "933838e2-bec1-440f-a634-9363c82e5b6d"
```
Copy the `Id` value.

**Option B: Microsoft Graph API**
```
GET https://graph.microsoft.com/v1.0/servicePrincipals?$filter=appId eq '933838e2-bec1-440f-a634-9363c82e5b6d'
```
Copy the `id` field.

### Step 2: Create the OIDC provider in ServiceNow

1. **All > System OAuth > Application Registry > New**.
2. For Yokohama and earlier: Choose **Configure an OIDC provider to verify ID tokens**.
   For Zurich and later: Select **New Inbound Integration Experience > New Integration > Third party ID token issued by OIDC supporting identity provider**.

| Field | Value |
|---|---|
| Name | Microsoft Entra ID |
| Provider name | Microsoft Entra ID |
| Client ID | `933838e2-bec1-440f-a634-9363c82e5b6d` |
| Active | Checked |

Under OAuth OIDC Provider Configuration, select existing Microsoft Entra ID or create new:

| Field | Value |
|---|---|
| Name | Microsoft Entra ID |
| OIDC Metadata URL | `https://login.microsoftonline.com/<tenantId>/v2.0/.well-known/openid-configuration` |
| OIDC Configuration Cache Lifespan | 120 |
| User Claim | `sub` or `oid` |
| User Field | User ID |
| Enable JTI Verification | Disabled |

Under Auth Scope, select `useraccount` and enable **Allow access only to APIs in selected scope**.

### Step 3: Create the ServiceNow integration user

1. **All > User Administration > Users > New**.

| Field | Value |
|---|---|
| User ID | Service principal object ID from Step 1 |
| Identity Type | Machine |
| Active | Checked |

2. Save.

### Assign roles to the integration user

Add: `catalog_admin`, `user_criteria_admin`, `user_admin`. If you used a custom role for the crawler account, add that too.

### Verification

- **OIDC provider**: Application Registry shows Microsoft Entra ID as Active with correct Client ID and metadata URL.
- **Integration user**: User exists with correct service principal object ID and roles.
- **Connector setup**: Select Federated Auth in M365 admin center and provide the ServiceNow instance URL.

## OAuth 2.0

Provision an OAuth endpoint in ServiceNow.

| Field | Recommended value |
|---|---|
| Name | Microsoft Search |
| Client ID | (autogenerated) |
| Client secret | (treat like a password) |
| Redirect URL | M365 Enterprise: `https://gcs.office.com/v1.0/admin/oauth/callback` — M365 Government: `https://gcsgcc.office.com/v1.0/admin/oauth/callback` |
| Active | Checked |
| Refresh token lifespan | 31,536,000 (one year) |
| Access token lifespan | 43,200 (12 hours) |
| Auth Scope | `useraccount` |

After connection, authenticate with a ServiceNow account that has at least the `knowledge` role.

## Microsoft Entra ID OpenID Connect

1. Register a single-tenant app in Microsoft Entra ID. No redirect URI needed.
2. Copy **Application (client) ID** and **Directory (tenant) ID**.
3. Create a client secret, save securely.
4. Retrieve service principal object ID:
   ```powershell
   Install-Module -Name Az -AllowClobber -Scope CurrentUser
   Connect-AzAccount
   Get-AzADServicePrincipal -ApplicationId "Application-ID"
   ```
   Or in Entra admin center: App registration > Overview > "managed application in local directory" > copy ObjectID.
5. In ServiceNow, register a new OAuth OIDC entity:

| Field | Value |
|---|---|
| Name | Microsoft Entra ID |
| Client ID | App (client) ID from Entra |
| Client Secret | Secret from Entra (replace the auto-generated ServiceNow one) |

6. Select search icon in **OAuth OIDC Provider Configuration** and create new:

| Field | Value |
|---|---|
| OIDC Provider | Microsoft Entra ID |
| OIDC Metadata URL | `https://login.microsoftonline.com/<tenantId>/.well-known/openid-configuration` |
| OIDC Configuration Cache Life Span | 120 |
| Application | Global |
| User Claim | `sub` |
| User Field | User ID |
| Enable JTI claim verification | Disabled |

7. Auth Scope → user account. Submit.
8. Create a ServiceNow user:

| Field | Value |
|---|---|
| User ID | Service Principal ID |
| Web service access only | Checked |

9. Assign `Knowledge` role.
10. Use App (client) ID and client secret in the admin center config wizard.

> **Note:** Don't turn on **Assignment required** on the enterprise app.

If using Advanced flow, enter the API namespace you created in ServiceNow.

Toggle **Rollout to limited audience** to specify users/groups. Then choose **Create**.

## Defaults after deployment

| Category | Setting | Default value |
|---|---|---|
| Users | Access permissions | Only people with access to the content in the data source |
| Users | Map identities | Data source identities mapped using Microsoft Entra IDs |
| Content | Query string | `active=true^workflow_state=published` |
| Content | Manage properties | See Manage properties |
| Sync | Incremental crawl | Every 15 minutes |
| Sync | Full crawl | Every day |

When status is **Ready**, validate by providing the `sys_id` of any knowledge article and verifying its user permissions.

## Customize settings

On the connector page in the admin center, choose **Custom setup**.

### Access permissions

- **Everyone**: all users see results.
- **Only people with access to this data source** (default): only authorized users.

**Note:** If an article and its KB have no `Can Read` user criteria, the article appears for everyone in the org provided `glide.knowman.block_access_with_no_user_criteria` is `false`. If `true` (or the service account can't read `sys_properties` — connector defaults to `true`), articles without user criteria are blocked.

**Warning (HRSD):** If KBs are in `sn_hr_core`, ensure the service account has `sn_hr_core.content_reader` or `sn_hr_core.admin`. Otherwise HR-scoped user criteria are filtered silently and restricted HR articles may be indexed as accessible to all users.

### Map identities

Default: email IDs to Entra UPN/Mail. Custom mapping formula available.

### Query string

Default: `active=true^workflow_state=published`. Use ServiceNow's encoded query string builder for custom filters.
